Honeycomb HTML

A description of the HTML for the Honeycomb project

The Doctype

<!DOCTYPE html>
<html class="no-js" lang="en">

Honeycomb is HTML5 and has the standard HTML5 doctype. The language is declared on the html tag. The no-js class is put on this tag as we use the modernizr script to control graceful degradation on older browsers.

The Head

<head>
    <meta charset="utf-8">
    <title>Website Template</title>
    <meta name="description" content="">
    <meta name="author" content="">
    <!--Microsoft. Delete if not required -->
    <meta http-equiv="cleartype" content="on">
    <meta http-equiv="X-UA-Compatible" content="IE=edge,chrome=1">
    <!-- Icons -->
    <link title="iPhone4" rel="apple-touch-icon-precomposed" sizes="114x114" href="img/ios-114.png">
    <link title="iPad" rel="apple-touch-icon-precomposed" sizes="72x72" href="img/ios-72.png">
    <link title="iPhone3" rel="apple-touch-icon-precomposed" href="img/ios.png">
    <link title="nokia" rel="shortcut icon" href="img/ios.png">
    <link title="Favicon" rel="shortcut icon" href="/favicon.ico">
    <!--iOS. Delete if not required -->
    <meta name="apple-mobile-web-app-capable" content="yes">
    <meta name="apple-mobile-web-app-status-bar-style" content="black-translucent">
    <!-- Mobile viewport optimized: j.mp/bplateviewport -->
    <meta name="viewport" content="width=device-width,initial-scale=1.0">
    <!-- CSS -->
    <link rel="stylesheet" href="css/style.css?v=1">
    <link rel="stylesheet" href="css/skin.css?v=1">
    <link rel="stylesheet" media="print" href="css/print.css?v=1">
    <!-- jQuery and HTML5 script so IE knows how to style the html5 elements -->
    <script src="js/libs/jquery.js"></script>
    <script src="js/libs/html5.js"></script>
</head>

The character set is the first thing in the document to be declared. This is utf-8. We chose this as it is the most universal character set available for our needs. The title, description and author are then declared. The next two tags are hacks for Microsoft Software:

  1. <meta http-equiv="cleartype" content="on"> - is for font smoothing on a Windows PC using ClearType technology.
  2. <meta http-equiv="X-UA-Compatible" content="IE=edge,chrome=1"> - forces Internet Explorer to render the page with it's latest version of the rendering engine. If the user has Chrome Frame installed then it will use it to render the page.

Next in the head are the declarations of the icons used for the page. The framework is built to work across multiple platforms and devices. Included in the head alongside the Favicon are desktop launch icons for iPad, iPhone, iPhone Retina and Nokia. The icons are declared as "precomposed", however if you want to use the apple shine effect then just remove the -precomposed bit at the end of the rel attribute. Also, there are several tags specifically for Apple devices. The first one apple-mobile-web-app-capable tells the device that this application can be used in standalone mode. This means you will not see the address bar at the top of the page and it will feel more like a native application. Only the status bar will appear at the top of the screen. It's style is then set by the next tag apple-mobile-web-app-status-bar-style. The "viewport" tag controls the scaling of the page when the device transitions from portrait to landscape view and vice versa. The Javascript plugin included at the end of the document - <script src="js/plugins/iOS-scale-fix.js"></script> will prevent any scaling when the view changes on an Apple device. More documentation

The CSS files are then called. The first - "style.css" contains resets, basic typography and other generic designs that form the basis of a site style. The "skin.css" conatins site specific style, i.e. colours, icons and other branding elements. The "style.css" should not be changed from one project to another (unless there are very specific needs). All front end development on a project should happen in "skin.css"

Also in the head are the Javascript libraries needed to render out the page. jQuery and html5. The rest of the javascript should be called from the bottom of the page. As the framework relies on jQuery most of the Javascript will be initiated in the document.ready function, and therefore does not need to run before the HTML is parsed. This increases page rendering speed as per Yahoo's recommendation

The Body

This is the generic HTML layout. This breaks the content into a header, page and footer. The page then has a content and sidebar:

<div id="wrap">

    <header id="header">
        <div class="headerWrap">
            <h1>Page Title</h1>
        </div>
    </header> <!-- end of header -->

    <div id="page" class="clear">
        <div class="pageWrap">
            <div id="content" role="main">
                <div class="contentWrap">
                    <section class="module">
                    </section> <!-- end of section -->
                </div>
            </div> <!-- end of content -->
            <div id="sidebar">
                <div class="sidebarWrap">
                    <section>
                    </section> <!-- end of section -->
                </div>
            </div> <!-- end of sidebar -->
        </div>
    </div> <!-- end of page -->

    <footer class="clear">
        <div class="footerWrap">
        </div>
    </footer> <!-- end of footer -->

</div> <!-- end of #wrap -->

Each major section of the page has a wrapping div, ie. <headerWrap>. This will give the page higher adaptability to design. Inside these areas are <section> tags which denote the individual modules on the page. When integrating this structure into a backend, typically the layout script would include the code for the <header> and <footer>, however this may be in the form of a partial view script. The page and pageWrap will be declared as well, however the content and sidebar content would be pulled from the view scripts. There would be occasions where the layout for the site may change significantly from one page to another. In this case the class on the body tag can be used to change the layout of an individual page. For instance and landing page may have no header and footer, and only a one column on the page, maybe for a photo gallery. The PHP framework will issue the body tag with a class, and then these elements could be hidden with css. The view script for the homepage would then contain only a "content" div that could be styled using the body class in the selector. The generic classes, i.e. ".btn", ".clearfix" and ".subhead" will be described in the documentation of the CSS.

The Javascript

Most of the javascript for the page will be at the bottom of the page. This includes any tracking code for the site. The tracking code is commonly put inline, and this would be one of the few concessions to putting the Javascript inline on the page. First the plugin scripts are called. Five plugin scripts that are highly useful are included in this framework.

  1. log.js - A useful javascript logging tool. Used for Javascript development
  2. placeholder.js - Adds placeholder text into form elements on legacy browsers
  3. selectivizr.js - Emulates CSS3 pseudo-classes and attribute selectors in IE 6 - 8
  4. respond.js - Adds Media Query support for IE 6 - 8
