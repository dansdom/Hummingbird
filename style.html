Honeycomb CSS

There are two CSS files associated with each project (there is also a print style which contains basic print styling, but it is not anticipated that this should be included in most of our projects).

  1. style.css
  2. skin.css

Style.css contains all the resets, general typography, form layouts, core page layouts including the grid system and device specific resets. This file is 'static' and should not need to be changed at all. Skin.css will contain styles to overwrite the default/reset style, helpers class styles, module and page specific style, plugin styles and the media rules. Skin.css contains a basic style which will be overwritten as needed to suit each new project.

Style.css

The first block of code is the reset css. This is based on Eric Meyer's Reset Revisited and includes HTML5 elements. One interesting line in this code not in Eric's reset is: html>body img {*-ms-interpolation-mode:bicubic} which improves the rendering of "stretched" images on IE7. The form elements have a reset section here too.

Below this is the typographical base. Basic styles for the headers, quotes, list styles and tables are defined here. The headers and body text are based on the Typographical Scale with the body text being set to 'pica'.

Next is the core layout of the page followed by the grid. If there was one thing that might be changed on our projects would be this layout (if the grid system was not being used). On screens with a maximum width of 620px all grid elements lose their floats and width values, thus stacking on top of each other.

There are two types of form layouts in style.css. The regular form places labels alongside the input elements. This form layout is for a form inside of a large block. The second layout is for narrow forms and mobile layouts. In this form the labels are not floating and sit above the input elements. Simply put on the class 'form-stack' to the form elements for this layout. Both of these are described in style.css. There are also several helper classes for the forms. Inputs can have the class 'small' or 'large'. Error fields that use the validation plugin will pick up either the class 'error' or 'help'. Other classes for form inputs include: 'submitField', 'fieldBox', 'radioField', 'checkField' and 'fieldLabel'.

The media rules at the bottom of this file are for general layout when using the grid system on a mobile device and the form elements on small screens. These should not be elaborated on. All other media CSS should be put at the end of skin.css.

Skin.css

The top of this document is where any external resources should be located. e.g. all fonts will be embedded here. Our preference is to use Google Fonts if we can otherwise use the Font Squirrel Generator to create the code to embed the font.

Site wide CSS such as the document background and link colours are next. This section should include the anchor suedo-classes as well as any other site wide suedo-classes or overwrites of the resets in style.css.

Each section after this contains either modules, page style or generic classes used for the site. Each module/page should be seperated by a blank line for easy scanning. Generally each selector in the module should start with the same element so that specificity is maintained and also to improve readability of the code. Each section should be commented as well. This applies to all sections in the CSS.

There are some general classes that should be mentioned here. These cover elements whose markup may change depending on their role, but their style will be common. These elements will share a common class. For example: every button in a site may look the same, however some of these buttons may be anchor tags, and some of them may be submit buttons in forms. These will all have the same class. Our convention is to use the class 'btn'. On top of this there may be different states of the button. A 'hover', 'disabled', 'small' or 'error' state may be needed for this button. In this case an extra class will go onto the element, and styled thus.

Once all the modules and pages for the site are declared, then the styles for the Javascript plugins are the declared. These should be treated in the same way as other page modules. Each plugin style should be commented and separated by white space to increase readability.

All CSS3 transitions should be declared in one place. These would go under the plugin style.

Last but not least are the media rules for different screen sizes and resolutions. These are put at the end of the document so that browsers which don't understand @media rules will be able to render the site without needing them. The smallest screen layout goes first followed by the next all the way up to the largest. Each larger size will over-write the last one. Iphone retina display comes last. The retina display will use mainly the 320 and 480 layouts and is mainly used to define high resolution images and thus needs to be after these rules.
